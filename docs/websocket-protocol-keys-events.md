# سند تحلیلی رویدادها و کلیدهای تبادل پروتکل وب‌ساکت (WebSocket Keys & Events Atlas)

این سند مرجع تخصصی برای هوش مصنوعی توسعه‌دهنده کدهای میکروکنترلر ESP32 است. در این سند، تک‌تک کلیدها (`keys`) و ساختار پیام‌هایی که در اثر اقدامات کاربر در داشبورد (نظیر تغییر فونت، جابه‌جایی ماژول‌ها، تغییر رنگ‌پلت، حذف یا ایجاد سگمنت‌ها و گروه‌ها) رخ می‌دهد، با ذکر تحلیل رفتاری و مثال‌های دقیق تحلیل شده است.

داشبورد هخامنشی به کلاینت‌ها آزادی عمل کامل می‌دهد تا چیدمان را تغییر دهند. برای همگام‌ساز ساختار چیدمان وب و حافظه فلش ESP32 (فایل `config.json` روی LittleFS)، از این کلیدها استفاده می‌شود.

---

## ۱. جدول مرجع کلیدهای رویداد اصلی (Event Keys Directory)

تمامی پیام‌ها حاوی یک کلید ریشه به نام `"event"` هستند که نوع رفتار پیام را مشخص می‌کند:

| کلید رویداد (`event`) | جهت ارسال (Direction) | توصیف رویداد و محرک فیزیکی |
| :--- | :--- | :--- |
| `"sync_layout"` | داشبورد $\rightarrow$ ESP32 | ارسال کل ساختار جدید چیدمان (ترتیب گروه‌ها، سگمنت‌ها، حالت‌ها و فونت) جهت ذخیره روی حافظه فلش |
| `"preferences_update"` | داشبورد $\rightarrow$ ESP32 | تغییر در ظاهر بورد نظیر رنگ شاخص، قلم پیشخوان یا انیمیشن‌ها |
| `"pin_control"` | داشبورد $\rightarrow$ ESP32 | تغییر وضعیت یک سوییچ یا ارسال پالس شستی فیزیکی به پایه‌ها |
| `"pin_status"` | ESP32 $\rightarrow$ داشبورد | فیدبک تاییدیه یا سیگنال وضعیت لحظه‌ای یک پایه فیزیکی به داشبورد |
| `"telemetry"` | ESP32 $\rightarrow$ داشبورد | ارسال گزارش دوره‌ای مقادیر خوانده شده از سنسورهای چهارگانه باستانی |
| `"device_ready"` | ESP32 $\rightarrow$ داشبورد | اعلام مشخصات بوت فیزیکی دستگاه (اسم، فریموور، رام) به محض اتصال سوکت |

---

## ۲. تحلیل سناریوهای پویای داشبورد و پروتکل انتقال داده

### سناریو الف: کاربر قلم (فونت) داشبورد یا تم را تغییر می‌دهد
وقتی کاربر در منوی تنظیمات، فونت شاهنشاهی بورد را مثلاً به حالت حماسی «لاله‌زار» تغییر می‌دهد یا میزان پدیداری کتیبه میخی را عوض می‌کند، داشبورد به سرعت پیام بروزرسانی تنظیمات را تحت رویداد `"preferences_update"` صادر می‌نماید.

**چرا ESP32 به این داده نیاز دارد؟**
ESP32 باید این مقادیر را در ساختار ترجیحات خود در حافظه ذخیره کند تا در دفعات بعدی گشایش صفحه وب توسط هر مرورگر دیگری، قالب بصری دقیقاً با همان سلیقه لود شود (میکروکنترلر به عنوان وب‌سرور، فایل پیکربندی یکپارچه را تحویل مرورگرها می‌دهد).

**JSON پیام ارسالی از داشبورد به ESP32:**
```json
{
  "event": "preferences_update",
  "payload": {
    "theme_mode": "dark",
    "accent_color_3": "#D4AF37",
    "accent_color_4": "#10B981",
    "font_family": "lalezar",
    "animations_enabled": true,
    "header_title": "سامانه هوشمند پادشاهی تخت جمشید",
    "cuneiform_opacity": 0.15,
    "cuneiform_color": "accent3"
  }
}
```

**توصیف کلیدهای بدنه:**
- `font_family`: حاوی مقادیری چون `"vazir"` (ساده)، `"lalezar"` (حماسی)، `"cairo"` (مدرن)، `"mono"` (فنی) است.
- `accent_color_3`: کد رنگ RGB طلایی انتخابی کاربر برای بخش‌های فلزی و کتیبه‌ها.
- `accent_color_4`: کد رنگ RGB سبز/فیروزه‌ای برای موقعیت‌های موفقیت و سنسورهای آب.
- `cuneiform_opacity`: عدد اعشاری بین `0.01` تا `0.40` جهت تعیین محو شدگی سنگ‌نگاره.

---

### سناریو ب: کاربر چیدمان یا ترتیب یک سوییچ را عوض می‌کند (Drag & Drop)
داشبورد مجهز به هسته چیدمان کشیدنی هخامنشی است. اگر کاربر گروه "سنسورهای پیرامونی" را به بالای گروه "فرماندهی" منتقل کند یا جای سوییچ چراغ آپادانا را با پمپ حوضچه عوض کند، ترتیب آرایه باید سنکرون شود.

**JSON پیام ارسالی از داشبورد به ESP32 تحت رویداد `"sync_layout"`:**
```json
{
  "event": "sync_layout",
  "payload": {
    "layout": {
      "groups_order": ["سنسورهای پیرامونی", "بخش فرماندهی"],
      "groups_cols": 1,
      "group_configs": {
        "بخش فرماندهی": { "maxCols": 3 },
        "سنسورهای پیرامونی": { "maxCols": 4 }
      }
    }
  }
}
```

**توضیحات فیلدها:**
- `groups_order`: ترتیبی است که گروه‌ها بر روی صفحه نمایش داده می‌شوند. هوش مصنوعی سخت‌افزار با جابه‌جای کردن این رشته‌ها در بدنه JSON متوجه چینش پیش‌فرض نهایی ستاپ می‌شود.
- `groups_cols`: تعادل چندستونه گرید پیشخوان روی کامپیوتر (مثلاً چند ستونه بودن کلی بورد).

---

### سناریو ج: کاربر یک سگمنت یا گروه جدید می‌سازد یا حذف می‌کند
اگر کاربر پایه جدیدی (مانند GPIO 13 برای "کنترل درب اصلی") بسازد، یا سگمنت بی‌مصرف روی پایه ۱۲ را به داخل سطل آشغال متحرک بیندازد، ساختار جدید سگمنت‌های متصل با متدهای زیر به حافظه میکرو پاس داده می‌شود.

**پیام سنکرون‌ساز سگمنت‌ها:**
```json
{
  "event": "sync_layout",
  "payload": {
    "segments": [
      {
        "id": "control_lamp",
        "type": "gpio_toggle",
        "pin": "2",
        "title": "روشنایی تالار آپادانا",
        "group": "بخش فرماندهی",
        "mode": "switch"
      },
      {
        "id": "door_shasti",
        "type": "gpio_toggle",
        "pin": "13",
        "title": "شاسی زنگ کاخ داریوش",
        "group": "بخش فرماندهی",
        "mode": "push"
      }
    ]
  }
}
```

**توصیف کلیدهای بهینه‌ساز سگمنت:**
- `id`: رشته تصادفی و تکرارناپذیر یونیک شناسه‌سازی ماژول.
- `pin`: شماره فیزیکی پایه GPIO میکروکنترلر که برای کنترل تحریک در نظر گرفته شده.
- `title`: نام فارسی دلخواه که کاربر روی سوییچ حک کرده است.
- `mode`: دو حالت دارد:
  1. `"switch"`: سوییچ قطع و وصل طولانی‌مدت (روشن/خاموش باقی می‌ماند).
  2. `"push"`: حالت شستی یا شاستی لحظه‌ای (فقط تا زمانی که دست کاربر روی سوییچ در گوشی یا لپ‌تاپ فشرده شده، پایه فعال است و به محض رها کردن سریعاً خاموش می‌شود).

---

### سناریو د: فرامین کنترل پایه‌ها فیزیکی بر اساس نوع مود سوییچ (`mode`)
مهم‌ترین رویداد عملیاتی، `"pin_control"` است. رفتاری که ESP32 بر اساس متغیر `mode` باید نشان دهد بدین ترتیب است:

#### ۱. اگر حالت سگمنت `"switch"` باشد:
کاربر یکبار کلیک می‌کند $\rightarrow$ پیام ارسال می‌شود $\rightarrow$ پایه خروجی روشن یا خاموش ثابت می‌گردد.

**نمونه پیام انتقالی:**
```json
{
  "event": "pin_control",
  "payload": {
    "pin": "2",
    "state": true
  }
}
```

#### ۲. اگر حالت سگمنت `"push"` (شستی لحظه‌ای) باشد:
به محض اینکه کاربر دست خود را روی دکمه می‌فشارد (رویداد `onPointerDown`/`onTouchStart`) پیام فورا ارسال می‌شود:

**پیام زمان فشرده شدن:**
```json
{
  "event": "pin_control",
  "payload": {
    "pin": "13",
    "state": true
  }
}
```
*میکروکنترلر پایه ۱۳ را HIGH می‌کند.*

به محض اینکه کاربر دستان خود را از روی کلید در فرانت به هر علتی برمی‌دارد (رویداد `onPointerUp`/`onTouchEnd`), پیام دوم ارسال می‌شود:

**پیام زمان رها ساختن دکمه:**
```json
{
  "event": "pin_control",
  "payload": {
    "pin": "13",
    "state": false
  }
}
```
*میکروکنترلر بلافاصله پایه ۱۳ را LOW می‌کند و موج پالس لبه پایینی کامل می‌شود.*

## ۳. نحوه ذخیره اطلاعات پیکربندی توسط ESP32 روی LittleFS
سورس کدی که روی ESP32 بارگذاری می‌شود، باید پیام‌های رویداد `"sync_layout"` و `"preferences_update"` را دریافت کرده و ساختار دریافتی را بر روی فایل `/config.json` در حافظه فلاش ذخیره کند. 

در زیر منطق سودوکد موردنیاز به زبان آردوینو آمده است تا هوش مصنوعی توسعه‌دهنده سخت‌افزار بتواند آن را در فرآیند ذخیره‌سازی ادغام کند:

```cpp
#include <LittleFS.h>
#include <ArduinoJson.h>

void saveWebSocketConfig(JsonObject incomingData) {
  // ۱. باز کردن فایل پیکربندی قدیمی روی فلاش
  File configFile = LittleFS.open("/config.json", "r");
  JsonDocument doc;
  
  if (configFile) {
    deserializeJson(doc, configFile);
    configFile.close();
  }
  
  // ۲. بروزرسانی و ادغام فیلدهای جدید واصل شده بر روی دیتای قبلی
  if (incomingData.containsKey("layout")) {
    doc["layout"] = incomingData["layout"];
  }
  if (incomingData.containsKey("segments")) {
    doc["segments"] = incomingData["segments"];
  }
  if (incomingData.containsKey("preferences")) {
    doc["preferences"] = incomingData["preferences"];
  }
  
  // ۳. نوشتن مجدد سند ترکیب یافته بر روی حافظه فلاش
  configFile = LittleFS.open("/config.json", "w");
  if (configFile) {
    serializeJson(doc, configFile);
    configFile.close();
    Serial.println("[System] فایل config.json روی LittleFS بروزرسانی شد.");
  } else {
    Serial.println("[Error] عدم دسترسی به نوشتن روی رام سخت‌افزار.");
  }
}
```

با اتکا به این سناریوها و کلیدهای بهینه‌سازی شده، ارتباط سخت‌افزار و داشبورد کاملاً زبان یکدیگر را فهمیده و هماهنگی ۱۰۰ درصدی بصری برقرار می‌گردد.
