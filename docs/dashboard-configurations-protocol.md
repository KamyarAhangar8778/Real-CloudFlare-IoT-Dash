# سند معماری و ساختار فایل‌های تنظیمات داشبورد (Dashboard Two-Phase Configuration Specs)

این مستند تشریح‌کننده فرآیند ارتباطی و ساختار داده دو هاب تنظیماتی جداگانه (فایل اول: تنظیمات ساختاری و بصری پایدار / فایل دوم: فرکانس و وضعیت‌های پویا سخت‌افزار) در لایه نِت است. 

به منظور ایجاد یک سناریوی کاربری بی‌نقص (UX) و جلوگیری از پرش‌های تصویری (Layout Shift) و هدررفت پهنای باند، معماری لود و راه‌اندازی داشبورد هخامنشی به صورت **دو مرحله‌ای (Two-Phase Hydration)** پیاده‌سازی شده است.

---

## نمودار توالی جریانی راه‌اندازی (Initialization Sequence)

```
[مرورگر کاربر/داشبورد]             [سرور مرکزی / گیت‌وی]             [حافظه فلاش سخت‌افزار/ESP32]
         |                                |                                   |
         | ---- (درخواست لود اولیه) ----->|                                   |
         |                                | <--- (فایل تنظیمات مرجع/پایه) -----|
         | <--- [فایل ۱: ساختار چیدمان] - |                                   |
         |    (ترسیم اسکلت و المان‌ها)      |                                   |
         |                                |                                   |
         |                              (برقراری لینک وب‌ساکت با گره سخت‌افزاری) |
         |                                | <--- (گزارش وضعیت پویای فعال) -----|
         | <--- [فایل ۲: مقادیر زنده] ----|                                   |
         |     (تزریق وضعیت و فعال‌سازی)    |                                   |
         v                                v                                   v
```

---

## مزایای مهندسی تفکیک داده‌ها (Separation of Concerns Benefits)

1. **پهنای باند بهینه (Optimal Payload Size):** اطلاعات سنگین ساختاری (مانند اسامی فارسی، نوع رله، نام قلم‌ها، آیکون‌ها و رنگ‌های هدر) فقط یک‌بار در فاز نخست واصل می‌شوند. پیام‌های بعدی تنها حاوی چند بایت دیتا (مقدار سنسور یا حالت باینری فعال) هستند.
2. **پایداری رابط کاربری (No Layout Thrashing):** ظاهر کلی گرید، تعداد سگمنت‌ها و فریم‌های کلود برای کاربر ترسیم می‌شود و صفحات سوسو (Flicker) نمی‌زنند.
3. **مکانیزم بازیابی سریع آفلاین (Offline-first Hydration):** در شرایطی که ارتباط اینترنت با سخت‌افزار نوسان داشته باشد، داشبورد ظاهر خود را حفظ کرده و فقط دکمه‌ها را به حالت "در حال اتصال به منبع" (Pending/Disabled) در می‌آورد تا از صدور فرامین موهوم جلوگیری شود.

## ۱. فایل اول: فایل پیکربندی مرجع و چیدمان بصری (Primary Layout & Design Blueprint)

این فایل بلافاصله پس از اولین دسترسی وب به داشبورد ارسال می‌شود. این فایل حاوی متادیتایی است که کمتر تغییر می‌کنند؛ مانند ساختار سگمنت‌ها، گروه‌بندی‌ها، موقعیت هدر، فونت و تم رنگی فعال.

* **نام پیش‌فرض فایل فیزیکی روی فلش:** `primary_layout_config.json`
* **بسامد بروزرسانی:** بسیار کم (فقط زمانی که کاربر چیدمان را تغییر دهد یا تنظیمات رنگ و فونت جدیدی اعمال کند).

### ساختار فیلدهای کلیدی (Schema Description)

* **`preferences`:** تنظیمات هویت بصری هخامنشی.
  * `theme_mode`: حالت پس‌زمینه (`"dark"` یا `"light"`).
  * `accent_color_3`: رنگ فلزی طلایی بورد.
  * `accent_color_4`: رنگ کمکی فرکانس سبز.
  * `font_family`: فونت فعال داشبورد که در سراسر بخش‌ها لود می‌شود.
  * `animations_enabled`: حالت انیمیشن پویا یا ساکن جهت کاهش پردازش سخت‌افزار.
  * `header_title`: عنوان سفارشی شده بالای صفحه.
  * `header_position`: قرارگیری افقی هدر (`"top"`) یا ستونی سمت چپ (`"left"`).
  * `cuneiform_opacity` / `cuneiform_color`: مانیتورینگ شفافیت و رنگ خط میخی پس‌زمینه کاخ پارسه.
* **`layout`:** نقشه پیوندهای گرید.
  * `groups_order`: چیدمان و اولویت از راست به چپ/بالا به پایین ستون‌ها.
  * `groups_cols`: تعداد کلی ستون‌های فعال در پیشخوان.
* **`segments_definition`:** شناسنامه سخت‌افزاری گره‌ها.
  * `id`: شناسه یکتا و یونیک هر ماژول برای شناسایی روی وب‌ساکت.
  * `type`: نوع سنسور یا رفتار عملکردی (مثال: `"gpio_toggle"`).
  * `pin`: مشخص کردن اینکه ماژول به کدام پایه فیزیکی میکروکنترلر متصل است.
  * `title`: نام فارسی دلخواه کاربر برای نمایش روی نمایشگر.
  * `group`: نام گروهی که این کارت متعلق به آن است.
  * `mode`: رفتار خروجی (`"switch"` برای روشن/خاموش پایدار، یا `"push"` برای شستی‌های تحریک لحظه‌ای پالس).

## ۲. فایل دوم: فایل وضعیت‌های پویای لحظه‌ای (Dynamic Runtime State Configuration)

پس از ترسیم رابط کاربری از روی فایل اول، داشبورد بلافاصله این فایل دوم را جهت بهینه‌سازی نهایی و بارگذاری آخرین وضعیت‌های پایه‌ها فراخوانی می‌کند. این فایل حاوی جزییات باینری رله‌ها، مقادیر آنالوگ سنسورها و جریان‌های اندازه گیری شده فعال در سخت‌افزار است.

* **نام پیش‌فرض فایل فیزیکی روی فلش:** `runtime_state_config.json`
* **بسامد بروزرسانی:** بسیار زیاد (هر چند ثانیه یکبار از طریق تلمتری وب‌ساکت یا به محض تغییر حالت رله‌ها توسط فرامین کاربر).

### ساختار فیلدهای کلیدی (Schema Description)

* **`module_states`:** آخرین مقادیر و ولتاژهای اعمال شده روی گره‌ها.
  * کلید ریشه هر شیء، همان شناسه یکتای سگمنت (`id`) است که در فایل اول جفت شده است.
  * `state`: آخرین وضعیت منطقی رله فیزیکی (`true` به معنای وصل / `false` به معنای قطع).
  * `last_value`: مقدار عددی ذخیره شده دیمرها یا پالس‌ها در رام سخت‌افزار (مثلاً شدت پالس PWM از `0` تا `255`).
  * `current_ma`: جریان مصرفی عبوری گزارش شده از لایه آمپراژ دستگاه جهت بررسی سلامت رله فیزیکی.
* **`sensors_data`:** آخرین کالیبراسیون سنسورهای باستانی.
  * `atarTemp`: دمای مطلق تالارها.
  * `anahitaMoisture`: رطوبت خاک گلدان‌ها و باغچه‌ها.
  * `vayuAirQuality`: خلوص مجاری ذرات معلق هوا.
  * `mithraLight`: لوکس نوری تابانیده شده بر سنسور مهر.
* **`device_telemetry`:** اطلاعات سلامتی هسته میکروکنترلر.
  * `uptime_seconds`: مجموع مدت زمان روشن بودن پیوسته دستگاه به ثانیه.
  * `rssi`: توان سیگنال شبکه بی‌سیم محلی وای‌فای (کیفیت ارتباط فیزیکی).

### نمونه واقعی فایل JSON دوم: `runtime_state_config.json`

```json

  "event": "runtime_state_config",
  "payload": {
    "module_states": {
      "module_1": {
        "state": true,
        "last_value": 185,

      },
      "module_2": {
        "state": false,
        "last_value": 0,
        "current_ma": 0.0
      },
      "module_3": {
        "state": false,
        "last_value": 0,
        "current_ma": 0.0
      }
    },
    "sensors_data": {
      "atarTemp": 25.8,
```

## ۳. نحوه اتصال، ادغام و بروزرسانی مستمر روی مرورگر

فرآیند ترکیب این دو منبع داده به شرح زیر روی مرورگر انجام می‌گردد:

1. **گام اول (برداشتن استخوان‌بندی):** داشبورد بارگذاری می‌شود و فایل اول (`primary_layout_config.json`) را تماشا می‌کند. به تعداد سگمنت‌های آرایه، کادرها و کارت‌های خام بدون دکمه فعال (Skeleton States) طراحی می‌شوند. فونت و رنگ بورد فوراً لود می‌گردند تا کاربر حس سرعت را تجربه کند.
2. **گام دوم (Hydration):** وب‌ساکت متصل شده و فایل دوم (`runtime_state_config.json`) واصل می‌گردد. کلاینت با مراجعه به فیلد شناسه (`module_1` و غیره) وضعیت سوئیچ‌ها را فورا به حالت متناظر تغییر می‌دهد و سنسورها را با انیمیشن‌های ملایم تابان زنده می‌کند.
3. **گام سوم (Sync مستمر):** از این پس با هر کلیک کاربر، پیام خرد `"pin_control"` با بدنه بسیار سبک مخابره شده و پس از بازگشت تاییدیه فیزیکی `"pin_status"` یا بسته تلمتری دوره‌ای، تنها فیلدهای متناظر در دیتای دوم تغییر می‌یابند و نیازی به دریافت مجدد کل چیدمان بورد در فایل اول نیست.

این معماری پایداری کامل فریم‌ریت، مهار مصرف ترافیک اینترنت و کارکرد منعطف سخت‌افزاری گره‌های مرزی پاسارگاد را تضمین می‌نماید.
