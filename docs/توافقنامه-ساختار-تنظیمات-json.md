# سند توافق‌نامه ساختار فایل‌های تنظیمات JSON (IoT Dashboard Schema Specs)

این مستند توافق بین لایه فرانت‌اند داشبورد، سرور بک‌اند (Cloudflare Work + Durable Objects + KV) و سخت‌افزار میکروکنترلر (ESP32) به منظور تعریف یکپارچه و بهینه ساختار داده‌های پیکربندی با فرمت JSON را مشخص می‌کند.

---

## ۱. معماری کلی داده‌ها و تفکیک وظایف

برای بهترین بازدهی در سخت‌افزار تعبیه‌شده (Embedded Hardware) و کاهش ترافیک شبکه، داده‌های سیستم به صورت کاملاً مجزا مدیریت می‌شوند:

```
┌────────────────────────────────────────────────────────┐
│               Cloudflare KV (محلی و توزیع‌شده)          │
│  داده‌های سنگین و کم‌تغییر: تنظیمات قالب، هدر، موقعیت و ماژول‌ها │
│  فایل مرجع: primary_layout_config.json                 │
└───────────────────────────┬────────────────────────────┘
                            │
┌───────────────────────────▼────────────────────────────┐
│         Cloudflare Durable Objects (ذخیره‌ساز فوق‌سریع)       │
│  وضعیت باینری لحظه‌ای پین‌ها (State) و تنظیمات پر تغییر داشبورد   │
│  پیکربندی داینامیک: /pins                              │
└────────────────────────────────────────────────────────┘
```

---

## ۲. ساختار نهایی و بهینه فایل‌های تنظیمات

طبق آخرین هماهنگی‌ها، کلیه کدهای غیرضروری از جمله دیتای ریداوت سنسورها، خروجی‌های تلمتری اضافی و ماژول‌های آنالوگ به کل از ساختار خذف شده و ساختار داده صرفاً بر مبنای **سگمنت‌های خاموش و روشن (GPIO Toggle)** بازطراحی گردیده است.

### الف) فایل تنظیمات بصری پایدار (`primary_layout_config.json`) - ذخیره‌سازی در KV

این قالب مشخص‌کننده فونت، هدر، نوع کنترل‌ها و ساختار صفحات است که از طریق مسیر کلاود `/config` در دسترس قرار می‌گیرد.

```json
{
  "preferences": {
    "theme_mode": "dark",
    "accent_color_3": "#D4AF37",
    "accent_color_4": "#10B981",
    "font_family": "lalezar",
    "animations_enabled": true,
    "header_title": "سامانه هوشمند مدیریت تجهیزات هخامنشی",
    "header_position": "top",
    "cuneiform_opacity": 0.12
  },
  "layout": {
    "groups_order": ["بخش فرماندهی", "سرویس‌های رفاهی"],
    "groups_cols": 2
  },
  "segments_definition": [
    {
      "id": "module_1",
      "type": "gpio_toggle",
      "pin": 2,
      "title": "سیستم تهویه مرکزی",
      "group": "بخش فرماندهی",
      "mode": "switch"
    },
    {
      "id": "module_2",
      "type": "gpio_toggle",
      "pin": 13,
      "title": "شاسی قطع کمکی",
      "group": "بخش فرماندهی",
      "mode": "push"
    },
    {
      "id": "module_3",
      "type": "gpio_toggle",
      "pin": 4,
      "title": "روشنایی محوطه پارسه",
      "group": "سرویس‌های رفاهی",
      "mode": "switch"
    }
  ]
}
```

#### فیلدهای کلیدی:
* **`segments_definition.pin`**: مقدار پایه قرارگیری سیگنال خروجی که اکنون به عنوان نوع داده عدد صحیح (`int`) ذخیره می‌شود که برای پارس در سخت‌افزار بسیار بهینه است.
* **`segments_definition.mode`**: تعیین رفتار پورت خروجی.
  * `"switch"`: شبیه‌ساز سوئیچ باینری دائم (روشن/خاموش مداوم).
  * `"push"`: شبیه‌ساز شستی یا کلید برای تحریک لحظه‌ای (پالس).

---

### ب) خروجی فیلتر شده مخصوص سخت‌افزار (`GET /config/esp`) - ذخیره‌سازی در KV

برای کاهش چشم‌گیر مصرف رم میکروکنترلر، یک اندپوینت فیلتر شده طراحی شده که اطلاعات اضافی مثل متون فارسی، رنگ تیک‌ها و افکت‌های بصری را دور می‌ریزد و صرفاً استخوان‌بندی سخت‌افزاری موردنیاز برای آردواینو را پس می‌دهد:

```json
[
  {
    "id": "module_1",
    "pin": 2,
    "mode": "switch"
  },
  {
    "id": "module_2",
    "pin": 13,
    "mode": "push"
  },
  {
    "id": "module_3",
    "pin": 4,
    "mode": "switch"
  }
]
```

---

### پ) وضعیت پویای باینری پین‌ها (`Durable Objects`) - مسیر `/pins`

ذخیره مقدار زنده پایه‌ها بر روی Durable Object جهت بازیابی سریع فرامین هنگام ریبوت سخت‌افزار یا قطع اتصال اینترنت.

* **تغییر تک پین (`POST /pins/{pin_id}`):**
  * ورودی: `{"value": true}`
  * خروجی: `{"pin": 2, "value": true}`

> ⚠️ **توافق‌نامه نهایی بهینه‌سازی کلاینت (حذف پولینگ تناوبی):**
> طبق تصمیم نهایی و جهت سبک‌سازی حداکثری و جلوگیری از سربار فچ‌های تکراری روی شبکه، داشبورد به هیچ عنوان درخواست پولینگ تناوبی (مانند فچ هر ۳ ثانیه‌ای وضعیت پین‌ها) به سمت سرور ارسال نخواهد کرد. 
> فرانت‌اند مقدار اولیه وضعیت پورت‌ها را از روی کش محلی و ساختار تنظیمات خوانده و منحصراً زمانی که کاربر وضعیت یک سوئیچ یا شاسی را تغییر دهد، اقدام به ارسال درخواست تکی `POST` برای بروزرسانی مقدار آن پین بخصوص روی Durable Object فرستاده می‌شود. این فرآیند باعث بهینه‌سازی کم‌نظیر پهنای باند و پاسخ‌دهی بلادرنگ کلیدها خواهد شد.

---

## ۳. پیشنهادهای مهندسی ارتقا و بهینه‌سازی ساختار (Optimizations)

جهت سبک‌سازی، توسعه‌پذیری و افزایش سرعت در لایه پردازشی ESP32، موارد زیر به عنوان پیشنهادات فنی ارائه می‌گردد:

### ۱. مکانیزم دریافت تجمعی وضعیت (Batch State Recovery API)
**مشکل فعلی**: سخت‌افزار در زمان بالا آمدن ناچار است برای دریافت تک تک پین‌ها چندین درخواست HTTP مجزا ارسال کند (`/pins/2` و `/pins/13` و غیره). این عملیات با احتساب تاخیر (Latency) در وای‌فای، زمان بالا آمدن سیستم را تا ۲ ثانیه افزایش می‌دهد.
**راهکار پیشنهادی**: اندپوینت `/pins` بدون پارامتر، تمام وضعیت‌های فعال میکرو را در قالب یک شیء تک‌سطحی بازگرداند:
```json
{
  "2": true,
  "4": false,
  "13": false
}
```
* **مزیت**: بازیابی کل سیستم با یک درخواست HTTP منفرد و پارس آن در سریع‌ترین زمان ممکن با کتابخانه `ArduinoJson`.

---

### ۲. فشرده‌سازی حداکثری کلیدها (Nano-Protocol Option) برای `/config/esp`
اگر بر روی بورد خود با محدودیت جدی حافظه استاتیک (RAM) روبه‌رو هستید، می‌توان خروجی فیلتر شده مخصوص سخت‌افزار را به حالت فشرده (Alias Keys) نیز در آورد:
```json
[
  {"i":"m1","p":2,"m":"s"},
  {"i":"m2","p":13,"m":"p"}
]
```
* **راهنما**: `i` برای شناسه ماژول، `p` برای شماره پورت فیزیکی پین، `m` نشانگر مود کاربری (`s` اختصار switch و `p` اختصار push).
* **مزیت**: کاهش ۵۰ درصدی طول رشته‌های JSON لود شونده در بافر حافظه ESP32.

---

### ۳. پیش‌فرض‌های امن ری‌استارت (Safe-State Definition during Boots)
برای دکمه‌های نوع شستی (`push`)، مقدار وضعیت در Durable Object نباید در حالت روشن پایدار ذخیره بماند. به زبان فنی با ریبوت شدن برد، همواره وضعیت پایه‌هایی با `mode: "push"` به صورت خودکار به `false` یا سطح منطقی غیر‌فعال بازنشانی شود تا رله‌های کلیدی و جک‌های هیدرولیک بدون درخواست لحظه‌ای کاربر روشن ناگهانی نگردند.

---

### ۴. متادیتای هماهنگ‌سازی (Sync Version Tag)
اضافه کردن فیلد `version` به تنظیمات KV جهت متوجه شدن ESP32 از نیاز به دریافت مجدد پیکسل‌ها تنها در صورت اعمال تغییرات جدید از سمت مدیر. بدین ترتیب نیازی به پولینگ (Polling) مداوم بابت تغییر چیدمان نیست:
```json
{
  "version": 1740004251,
  "preferences": { ... }
}
```
بدین وسیله در صورتی که نسخه ذخیره شده در حافظه EEPROM برد با نسخه سرور برابر بود، سخت‌افزار از مرحله دانلود تنظیمات عبور کرده و فورا بالا می‌آید.
